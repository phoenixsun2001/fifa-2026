# AI 大数据统计预测功能 — 优化路线图

> 建立日期：2026-06-14
> 关联代码：`2026.tsx`（单文件 ~3695 行）
> 设计原则：先建"测量基础设施"，再动"模型"，最后做"产品"。没有可复现、可量化误差的基准，无法证明模型改进是否有效。

---

## 一、现状诊断

整个预测系统由三层构成，全部封装在 `2026.tsx`：

| 层 | 核心函数 | 位置 | 作用 |
|---|---|---|---|
| **单场引擎** | `simulateMatchRealistic` | `2026.tsx:752` | FIFA 排名差 → xG → 泊松采样 |
| **随机源** | `getPoissonGoal` (Knuth) | `2026.tsx:740` | 生成进球数 |
| **批量推演** | `handleBatchSimulation` | `2026.tsx:1299` | N=100~1000 次蒙特卡洛，统计出线/夺冠率 |
| **单次模拟** | `handleAutoSimulateOnce` | `2026.tsx:1163` | 单遍跑完整届杯赛（含淘汰赛） |

**单场引擎核心公式**（`2026.tsx:760-766`）：

```
homePower = 100 - rank + formBoost     (away 同值镜像)
powerDiff = homePower - awayPower
homeExp   = 1.25 + powerDiff × 0.012
awayExp   = 1.25 - powerDiff × 0.012   ← 完全对称镜像（核心缺陷）
```

然后各自独立泊松采样，加东道主/大区修正，clamp 到 [0.5, 3.5]。

---

## 二、核心科学瓶颈（按影响降序）

1. 🔴 **攻防能力未分离**：只有单一标量"整体实力"，进攻力与防守力是同一数的镜像。无法区分挪威（攻强守弱）与摩洛哥（守强攻弱）。
2. 🔴 **双泊松独立性假设**：两队进球独立采样，真实足球中负相关；系统性低估 0-0/1-1，错估 1-0/0-1（Dixon-Coles 数十年前的标准解法）。
3. 🟡 **蒙特卡洛样本与可信度**：默认 100 次、上限 1000 次；长尾事件（夺冠率）统计误差 ±20% 相对值；只给点估计、无置信区间；同步阻塞主线程。

## 三、优化方向全景（分层矩阵）

### A 层 — 模型引擎（准确率根基）

| # | 方向 | 当前 | 优化后 | 成本 |
|---|---|---|---|---|
| A1 | **攻防分离** | 单标量实力 | 每队 `attack`/`defense` 双维，λ=base×atck_i×dfns_j⁻¹ | 中 |
| A2 | **Dixon-Coles 模型** | 独立双泊松 | 双变量泊松 + τ 低分修正 + 负相关 | 中高 |
| A3 | **Elo 评级替代 FIFA 排名** | FIFA 排名（滞后+缺陷） | Elo（连续、自更新、业界主流） | 中 |
| A4 | **淘汰赛保守系数** | 淘汰赛=小组赛基线 | 淘汰赛 xG 下调 | 低 |
| A5 | **点球模型** | `0.5+排名差×0.003` 任意系数 | Elo 差距 + 门将维度 | 低 |
| A6 | **xG clamp 放宽** | [0.5, 3.5] 封顶 | 极端悬殊局能正确拉开 | 低 |

### B 层 — 蒙特卡洛与可信度

| # | 方向 | 当前 | 优化后 | 成本 |
|---|---|---|---|---|
| B1 | **Web Worker 化** | 同步阻塞主线程 | Worker 跑，主线程不卡 → 支持 10000+ 次 | 中 |
| B2 | **可种子 PRNG** | `Math.random()` 不可复现 | mulberry32 种子化，可复现/可分享 | 低 |
| B3 | **置信区间输出** | 仅点估计 | ±95% CI（二项分布），显示 `15%±3%` | 低 |
| B4 | **方差缩减** | 纯随机采样 | 对立变量 / 分层抽样，同等样本精度翻倍 | 中 |

### C 层 — 数据资产激活

| # | 方向 | 现状 | 优化 | 成本 |
|---|---|---|---|---|
| C1 | **激活阵容数据** | `teamSquads`（`2026.tsx:189`）存在但**完全没参与预测** | 球员俱乐部级别/位置派生攻防评分，反哺 A1 | 中 |
| C2 | **formBoost 可解释化** | 硬编码黑箱整数 | 拆成近期胜率/阵容质量/东道主/伤停，开放用户调整面板 | 中 |
| C3 | **历史交锋嵌入** | 无 | 静态嵌入近 N 场战绩、对战历史 | 高 |

### D 层 — 产品/交互（差异化与粘性）

| # | 方向 | 价值 |
|---|---|---|
| D1 | **情景模拟(What-if)** | "假设挪威小组出局，夺冠格局如何变"——预测产品最高粘性功能 |
| D2 | **预测溯源** | "阿根廷为何 18%"→ 关键胜负路径 + 球员贡献，打破黑箱 |
| D3 | **条件贝叶斯更新** | 复用已 locked 机制，随赛事推进让预测自动收敛 |

### E 层 — 工程质量（隐性技术债）

| # | 方向 | 现状 |
|---|---|---|
| E1 | **消除重复逻辑** | `handleAutoSimulateOnce` 与 `handleBatchSimulation` 各自重写了 ~140 行小组积分+淘汰赛逻辑，易分叉 |
| E2 | **引擎模块化** | 引擎/数据/UI 全挤在单文件 3695 行 |
| E3 | **核心函数单测** | 泊松均值、积分排序、R32 配对零测试 |

## 四、依赖关系总览

```
Phase 0 地基重构 ──► Phase 1 可信度 ──► Phase 2 准确率核心 ──► Phase 3 数据激活 ──► Phase 4 产品差异化
 (可测/可复现)        (可量化误差)        (单点击穿点)            (兑现大数据)          (有粘性)
   │                    │                    │                      │
   └─ E1/E2 抽引擎       └─ B1 Worker         └─ A1 攻防分离          └─ C1 阵容评分 ──► D2 溯源
   └─ B2 种子PRNG        └─ B3/B4 误差         └─ A2 Dixon-Coles          (天然落点)
   └─ E3 单测                                  └─ A3 Elo 评级 ◄── 反哺
```

**铁律**：Phase 0 是所有后续阶段的前置——它把 `simulateMatchRealistic` 从"耦合在 UI、不可测、不可复现"变成"独立纯函数、有测试、有种子"。之后每一阶段改进都能用同一回测基准验证。

## 五、分阶段路线图

### Phase 0 — 地基重构（可测 + 可复现）✅ 已完成 2026-06-14

**目标**：把引擎从 UI 剥离，建立可量化改进的基准线。技术最不性感、杠杆最大。

> **落地结果**：引擎已提取至 `src/engine/`（rng.ts / data.ts / sim.ts / tournament.ts），25 个单测全绿，
> `simulateTournament` 纯函数统一了**三处**重复的"小组+淘汰赛"逻辑（原 roadmap 只记两处，实际还有
> `handleJourneySimulation` 单队分析器，已用最小改动适配种子化签名）。
> 两个 handler 改为派生自 `simulateTournament` 结果。数据层镜像暂留 2026.tsx（行为中立的后续清理）。

| 任务 | 说明 | 方向 |
|---|---|---|
| 抽取纯函数 | `simulateTournament(prng, lockedMap, opts) → {standings, champion, ...}`，消除两处重复 | E1 |
| 引擎模块化 | 拆 `engine/poisson.ts` / `engine/sim.ts` / `engine/tournament.ts` | E2 |
| 核心单测 | 泊松均值收敛、积分排序稳定性、R32 配对正确性、边界 | E3 |
| 可种子 PRNG | `mulberry32(seed)` 替换 `Math.random()` | B2 |

- **交付物**：引擎独立模块 + 测试套件 + "种子 42 → 同一结果"
- **验收指标**：`simulateTournament` 覆盖率 > 85%；同种子 100 次结果完全一致

### Phase 1 — 可信度（蒙特卡洛专业级）✅ 已完成 2026-06-14

**目标**：让百分比从"玩具数字"变成"带误差棒的专业预测"。不提升准确率，但提升可信度。

| 任务 | 说明 | 方向 |
|---|---|---|
| Web Worker 化 | 批量推演迁 Worker，主线程不卡 → 支持 10000+ 次 | B1 |
| 置信区间 | 二项分布 ±95% CI，显示 `17.3% ± 2.1%` | B3 |
| 方差缩减 | 对立变量 / 分层抽样，同等样本精度翻倍 | B4 |
| 校准指标 | 引入 Brier Score / 对数损失作为自评 | 新增 |

- **交付物**："种子 + 大样本 + 误差棒"专业级推演
- **验收指标**：10000 次推演主线程 60fps；冠军率 CI 宽度 < ±2%（强队）

### Phase 2 — 准确率核心（🎯 单点击穿点）🔬 核心完成 2026-06-14

> **落地结果**：`simulateMatchRealistic` 从「独立双泊松采样」升级为 **Dixon-Coles 联合采样**
> （`sim.ts` 新增 `tau` τ低分相关性修正 + `dixonColesJoint` 归一化联合概率矩阵 +
> `sampleJoint` 单随机数采样；DC_RHO=−0.13）。修正了独立泊松对 0-0/1-1 的低估与对 1-0/0-1 的高估
> （核心缺陷 #2）。xG clamp 由 [0.5,3.5] 放宽至 [0.4,4.0]（A6）。4 个 DC 相关性单测通过，
> 全部 sanity 测（西班牙>新西兰 / 10+ 冠军 / 强队胜率>0.7）在模型切换后仍绿。
> **诚实延后**：A1 攻防分离需 Phase 3 阵容数据才有真实信号（否则造假分离=cargo-cult）；
> A4 淘汰赛保守系数 / A5 点球模型为低优先项。

**目标**：模型范式升级。全图最关键阶段，直接抬高准确率天花板。

| 任务 | 说明 | 方向 |
|---|---|---|
| **攻防分离** | `attack`/`defense` 双维，`λ = base × attack_i × defense_j⁻¹` | A1 |
| **Dixon-Coles** | 双变量泊松 + τ 低分修正 + 负相关 | A2 |
| clamp 放宽 | xG 上限放宽，极端悬殊局能正确拉开 | A6 |
| 淘汰赛系数 | 淘汰赛 xG 基线下调（更保守） | A4 |
| 点球模型 | Elo 差距 + 门将维度 | A5 |

- **交付物**：升级后单场引擎；可用历史赛果回测验证
- **验收指标**：Brier Score 较 Phase 1 显著下降；挪威 vs 摩洛哥踢出风格差异

### Phase 3 — 数据激活（兑现"大数据"）

**目标**：让已有但沉睡的数据参与预测。Phase 2 的 `attack/defense` 双维在此提供天然落点。

| 任务 | 说明 | 方向 |
|---|---|---|
| **激活阵容数据** | `teamSquads` → 前锋/中场贡献 attack，后卫/GK 贡献 defense | C1 |
| Elo 评级 | Elo 替代 FIFA 排名作为实力源 | A3 |
| formBoost 可解释化 | 拆子项，开放用户可调参数面板 | C2 |

- **交付物**：数据驱动攻防评分 + 可交互实力调节面板
- **验收指标**：移除/调整核心球员后预测有可解释变化

### Phase 4 — 产品差异化（从工具到产品）

**目标**：在坚实引擎之上，做有粘性、有差异化的体验。

| 任务 | 说明 | 方向 |
|---|---|---|
| 预测溯源 | "阿根廷为何 18%"→ 关键胜负路径 + 攻防贡献 | D2 |
| What-if 情景模拟 | "假设某队出局/伤核心"，看夺冠格局变化 | D1 |
| 条件贝叶斯更新 | 复用 locked 机制，随赛事推进自动收敛 | D3 |
| 历史交锋嵌入 | （可选，数据最重） | C3 |

## 六、关键决策点

| 决策 | 选项 | 建议 |
|---|---|---|
| Elo vs FIFA 排名 | 用哪个做实力源 | Elo 更优（连续、自更新），需初始化数据；Phase 3 引入 |
| Web Worker vs 分片 yield | 如何不卡 UI | Worker 彻底隔离，推荐 |
| 回测数据集 | 如何验证"变准了" | 用 2018/2022 世界杯淘汰赛结果做 calibration check |

---

## 七、极简路径

- **只做一个阶段**：Phase 0 + Phase 2 最小子集（抽引擎 + 攻防分离 + Dixon-Coles）。改动收敛在一个函数，但范式升级，全下游百分比一次性受益——"简单做到极致"的真正击穿点。
- **只做两个阶段**：加 Phase 1，让升级后的结果可量化、可信任。
